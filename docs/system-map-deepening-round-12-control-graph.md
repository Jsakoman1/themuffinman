# System Map Deepening — Round 12: Control Graph and Source-Reference Integrity

Date: 2026-07-22  
Status: **observed analysis**  
Scope: canonical documents, YAML references, generated contracts, work-plan graph, verifier, audits, and backlog/control linkage

## Executive finding

The repository has a real control system rather than a collection of informal
documents. `docs/work/*.yaml` plus `scripts/verify-work.rb` define completion
state, strict required paths, serial execution, evidence capture, and master/child
relationships. The agent operating model points to canonical API and intent
section files, and the frontend generator now consumes those references.

The principal control risk is graph scale and source heterogeneity: `docs/work`
contains historical plans, active queues, master plans, execution inventories,
and analysis plans. A filename or markdown status cannot be treated as truth.
The verifier is the completion authority, while capability inventory, runtime
matrix, target catalog, and audits remain separate layers with different
semantics.

This graph is healthy when every reference is resolvable, generated outputs are
fresh, one serial item is active, and final state is produced by the verifier.

## Canonical-source boundaries

### Implementation control

`docs/implementation-control.md` clearly states the active workflow, normal
states, strict path requirements, serial execution rules, runtime evidence rules,
and master-plan semantics. The verifier writes evidence including command,
timestamp, revision, exit code, output, and changed files.

### Agent operating model

`docs/agent-operating-model.yaml` is the safety/control root. Its API and intent
catalogs are external section references, not inline root data. The frontend
contract generator was corrected during the first analysis program to load those
referenced files before emitting endpoint and intent constants. Contract freshness
now proves that the generated artifact reflects the full referenced catalogs,
not only root YAML content.

### System map and analysis program

`docs/system-map.md` is the cross-system index. The first program has a verified
master; the second deepening program has a verified preflight and a planned master
with strict serial children. Round 8–11 documents and their work plans are now
verifier-backed.

### Capability/runtime/target layers

These layers are intentionally distinct:

- capability inventory: current implementation and evidence inventory;
- runtime acceptance matrix: browser/device evidence and pending runtime gaps;
- target capability catalog: intended production capability scope;
- runtime evidence JSON: concrete artifacts for selected scenarios;
- audits: diagnostics that detect drift or structural problems;
- work plans: controlled execution and completion state.

The separation is correct. The danger would be copying one layer's status into
another without evidence.

## Verifier and audit assessment

The verifier enforces useful invariants:

- strict plans must list exact required paths;
- required paths must change from the serial start snapshot;
- serial plans start and verify one inventory item in order;
- task validation commands cannot recursively invoke the verifier;
- master plans depend on independently verified children;
- evidence is generated by executed commands rather than handwritten claims.

The audit set covers backend, frontend, docs-as-tests, plan coverage, target
catalog, inventory freshness, runtime acceptance, native handoff, impact, API
contract drift, endpoint callsites, permissions, duplication, mappers, repository
fetches, UI entrypoints, and work-plan recursion.

The strongest mechanical checks currently passing for this program are:

- plan coverage and serial graph integrity;
- work-plan recursion audit;
- target capability catalog and inventory freshness;
- runtime acceptance status;
- generated frontend contract freshness;
- agent safety, frontend, backend, docs, native handoff, and impact validations
  as invoked by the completed rounds.

## Control weaknesses

1. **Historical plan density.** Many YAML plans remain in the work directory for
   history or prior programs. Generic searches can overcount active work. The
   active master and verifier evidence must remain the status source.
2. **Audit scope differs by command.** For example, plan coverage focuses on
   active capability inventory plans and master graph rules; it is not a semantic
   review of every historical analysis document.
3. **Generated artifact freshness is consumer-specific.** The frontend contract
   generator now follows agent section references, but other generated reports or
   derived inventories may have different source boundaries.
4. **No universal reference linter.** The control system audits many classes of
   integrity, but a single command does not yet verify every Markdown link, YAML
   path reference, source evidence path, generated artifact, and plan dependency.
5. **Uncommitted baseline.** Current analysis docs, plans, and the generator fix
   are uncommitted. Verifier evidence records the old revision plus changed files;
   this is valid evidence for the worktree but not a clean revision comparison.
6. **Backlog linkage is partly procedural.** The rules require stable IDs for
   deferred work, but analysis findings still need consistent mapping to a single
   persistent backlog item and eventual closure evidence.

## Integrity rules to preserve

- Do not mark a capability verified from a static route or catalog entry alone.
- Do not treat generated diagnostics as canonical status.
- Do not let a Markdown summary replace machine-readable active state.
- Do not add child plans outside the active master/inventory graph.
- Do not alter old migrations or infer runtime evidence from source tests.
- Record any new deferred work with a stable backlog ID before closing the round.
- Keep root/section YAML references explicit and test the generator against them.

## Recommended control improvements

1. Add a lightweight canonical-reference audit that resolves all declared YAML,
   Markdown, evidence, generated, and work-plan paths.
2. Add a plan-kind/status report distinguishing active, planned, verified,
   historical, and deferred plans without relying on filenames.
3. Add a source-to-generated registry for contract generators and derived reports.
4. Add a stable finding register linking each analysis finding to a backlog ID,
   capability ID, owner, and evidence requirement.
5. Record a clean post-analysis baseline before the next implementation batch,
   without committing or pushing automatically.

## Conclusion

Round 12 confirms that the control system is a substantive architecture layer.
Its core model is sound: YAML plans and verifier evidence govern completion,
while inventories, runtime matrices, target catalogs, and audits provide distinct
evidence and diagnostic views. The next maturity step is a universal reference
and finding registry that makes the remaining cross-layer graph mechanically
auditable.
