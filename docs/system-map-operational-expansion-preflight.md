# Operational System-Map Expansion Preflight

Status: prepared for goal pursuing. Reviewed: 2026-07-22.

## Readiness conclusion

The six-round program is ready for serial execution. It builds on verified truth,
interface, data, workflow, evidence, delivery, and drift registries. It must extend
those relationship indexes rather than copy capability status or manufacture runtime
and production claims.

## Required evidence distinctions

- `source-trace`: code, configuration, generated contract, or static registry path.
- `validated-contract`: focused test, audit, or verifier result.
- `runtime-proof`: current browser, device, provider, job, or operator artifact.
- `inferred-risk`: bounded review finding with source basis and uncertainty.
- `unknown`: deliberately unresolved fact, including unavailable external operations.

## Existing baseline signals

- Endpoint reconciliation currently classifies 214 endpoints: 140 Web-source-linked,
  4 admin/agent, 3 native-handoff, and 67 unclassified non-Web review rows.
- Persistence source contains 76 Flyway migrations, 8 ownership domains, and 6
  current workflow coverage rows.
- Runtime acceptance records 30 passed and 24 pending scenarios. A future
  observability registry must not turn either count into production-scale evidence.
- Delivery provenance validates four repository-visible build paths while CI,
  deployment, release, rollback, backup, and restore remain external unknowns.

## Plan improvements made by preflight

- Added a shared evidence and row-metadata protocol to the master.
- Added dependency and output fields to every inventory item.
- Made Round 37 require the actual generator script, Make command, and operating
  documentation rather than accepting a report-only closeout.
- Defined operational proof as a later evidence layer, preventing Rounds 35 and 36
  from being mistaken for production validation.

## Start protocol

Start Round 32 with the serial work-start command, create its report and registry,
verify only that item, and then proceed in inventory order. Any discovered feature
or operations work is recorded with a stable backlog ID; no round changes canonical
capability or runtime status without its owning evidence workflow.
